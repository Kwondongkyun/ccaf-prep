## EXPLAIN

> Task 1.2 — coordinator-subagent 패턴으로 멀티 에이전트 시스템 오케스트레이션

### 한 줄 정의

**Hub-and-Spoke** = 하나의 코디네이터 에이전트가 모든 서브에이전트 간 통신·에러 처리·정보 라우팅을 책임지는 구조.

| 항목 | 내용 |
|------|------|
| 비유 | 축구 감독(코디네이터) ↔ 선수(서브에이전트). 선수끼리 직접 작전 짜지 않고 감독이 다 통제. |
| 핵심 키워드 | hub-and-spoke, coordinator, subagent isolation, delegation |

### 구조 다이어그램

```
              ┌──────────────────────┐
              │     Coordinator      │
              │ - 작업 분해           │
              │ - 위임                │
              │ - 결과 집계           │
              │ - 갭 평가 → 재위임     │
              └──────────────────────┘
                  ▲ ▲ ▲ ▲
                  │ │ │ │  (모든 통신 hub 통과)
        ┌─────────┘ │ │ └─────────┐
        │           │ │           │
   ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
   │ 웹검색  │  │ 문서   │  │ 종합   │  │ 보고서 │
   │ Agent  │  │ Agent  │  │ Agent  │  │ Agent  │
   └────────┘  └────────┘  └────────┘  └────────┘
       ↑          ↑           ↑           ↑
       └─ 격리된 컨텍스트. 서로 직접 통신하지 않음 ─┘
```

### 4가지 핵심 사실

**1. 서브에이전트는 컨텍스트를 자동 상속하지 않는다 (함정!)**
- 코디네이터의 대화 이력은 서브에이전트에게 자동으로 넘어가지 **않음**.
- 코디네이터가 Task 도구로 서브에이전트를 호출할 때, **프롬프트에 명시적으로 컨텍스트를 포함**해야 함.
- 이 함정은 D1 시험에서 반복 출제. (다음 블록 1-3에서 더 깊이)

**2. 코디네이터의 4가지 역할**
- 작업 분해 (decomposition)
- 위임 (delegation) — 어느 서브에이전트에게 보낼지 결정
- 결과 집계 (aggregation)
- **갭 평가 + 재위임** (iterative refinement) — 종합 출력의 빈틈 발견 시 추가 쿼리 위임

**3. 모든 통신은 코디네이터를 거친다**
- 서브에이전트끼리 직접 통신 X.
- 이유: **관찰 가능성**, **일관된 에러 처리**, **정보 흐름 통제**.
- 서브에이전트 A의 결과를 서브에이전트 B에 넘기려면? → A가 코디네이터에 반환 → 코디네이터가 B의 프롬프트에 포함해 호출.

**4. 작업 분해의 함정 — 너무 좁게 쪼개면 커버리지 부족**
- 폭넓은 리서치 주제를 너무 좁은 하위 작업으로 쪼개면, 어느 서브에이전트도 큰 그림을 못 봄.
- 분해는 **균형** — 중복 최소화 + 충분한 커버리지.

### 코디네이터 작성 원칙

```
❌ 단계별 절차 프롬프트:
   "1. 웹 검색해. 2. 결과를 종합해. 3. 보고서 만들어."

✅ 목표·품질 기준 프롬프트:
   "이 주제에 대한 1차·2차 소스를 다양하게 수집하고,
    상충하는 주장은 별도로 표시하라.
    인용은 URL과 발췌를 함께 포함."
```

이유: 서브에이전트가 **자기 영역에서 적응**할 수 있어야 함. 단계별 절차는 적응성 죽임.

### 동적 위임 vs 고정 파이프라인

| 쿼리 종류 | 처리 방식 |
|---------|---------|
| "Python에서 dict와 OrderedDict 차이" | 코디네이터가 웹검색 1번만 호출. 종합 단계 skip. |
| "AI 윤리에 대한 종합 보고서" | 웹검색 + 문서분석 + 종합 + 보고서 = 풀 파이프라인 |

→ 코디네이터는 **쿼리 복잡도에 따라 동적으로** 어떤 서브에이전트를 호출할지 결정. 항상 풀 파이프라인 X.

### 반복적 정제 루프 (Iterative Refinement)

```
[1] 코디네이터 → 서브에이전트(웹·문서) 위임 → 결과 수신
       ↓
[2] 종합 서브에이전트 호출 → 종합 출력 수신
       ↓
[3] 코디네이터가 갭 평가 → "X 영역 커버 부족"
       ↓
[4] 검색·분석 서브에이전트에 타겟 쿼리 재위임
       ↓
[5] 종합 다시 호출 → 커버리지 충분 → 종료
```

→ 종합은 한 번이 아니라 **갭 만큼 반복**. 시험 시나리오에 자주 등장.

### 자주 나오는 함정

| 잘못된 생각 | 올바른 생각 |
|-----------|-----------|
| "서브에이전트끼리 직접 통신하면 빠르겠지" | 모든 통신은 코디네이터 hub 통과 |
| "한번에 풀 파이프라인 돌리자" | 쿼리 복잡도에 따라 동적 위임 |
| "코디네이터는 한 번 종합 호출하면 끝" | 갭 평가 → 재위임 → 종합 반복 |
| "서브에이전트가 코디네이터 컨텍스트 알아서 받겠지" | 명시적 프롬프트 포함 필수 |
| "서브에이전트는 단계별 절차로 지시" | 목표·품질 기준으로 작성 (적응성 보존) |

- **Q1: B (Hub-routed)** — Hub-and-spoke의 정의. 모든 통신은 코디네이터를 통과. 이유: 관찰 가능성·에러 처리·정보 흐름 통제. P2P (A 선택)는 안티패턴.
- **Q2: B (목표·품질 기준)** — **D1 함정 F1과 결.** 단계별 절차는 서브에이전트의 적응성을 죽임. 코디네이터는 "어떻게"가 아니라 "무엇을/왜"를 명시.
- **Q3: B (Iterative refinement)** — 코디네이터의 4가지 역할 중 "갭 평가 + 재위임". A는 비효율 (이미 한 작업 버림). C는 품질 포기. D는 갭이 외부 정보 부족이 아닐 때 잘못된 선택.

### 출제 변형

- **"코디네이터는 항상 풀 파이프라인을 돌려야 한다?"** → ❌. 쿼리 복잡도에 따라 동적 위임. "오늘 날씨" → 검색만, 종합·보고서 skip.
- **"서브에이전트끼리 메모리 공유?"** → ❌. 격리된 컨텍스트. 결과 공유는 코디네이터 경유.
- **"갭이 발견되면 사용자에게 즉시 에스컬레이션?"** → ❌. 먼저 재위임으로 자율 보강. 진전 불가일 때만 에스컬레이션 (D5 Task 5.2).
